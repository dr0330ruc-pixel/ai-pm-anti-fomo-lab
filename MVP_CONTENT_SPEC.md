# MVP_CONTENT_SPEC.md

> 版本：v0.1  
> 日期：2026-06-07  
> 适用范围：第一版网页 / H5 MVP  
> 内容原则：只使用静态、可人工维护、无需登录和无需后台的数据；不复制付费内容，不声称来自内部资料。

## 1. 内容设计目标

本文件用于定义第一版网页 / H5 MVP 需要展示和使用的内容资产，保证产品能跑通以下闭环：

```text
能力自测 -> 结果解释 -> 推荐训练任务 -> JD 翻译 -> 作品提交 -> Rubric 评分反馈
```

本文件解决的问题：

- 第一版需要哪些静态内容。
- 自测题、计分规则、训练任务、JD 翻译模板和作品 Rubric 如何互相连接。
- 页面开发前，前端可直接配置哪些文案、选项、分值和模板。
- 半人工作品点评时，应该依据什么标准给用户反馈。

本文件不解决的问题：

- 不设计新页面。
- 不设计后台、登录、支付、数据库或用户历史记录。
- 不设计自动抓取 JD、自动抓取新闻或自动化精准评分系统。
- 不提供付费课程内容、内部招聘资料或不可引用的私有信息。
- 不承诺第一版能自动生成完整作品或替代人工点评。

P1 首页忽略榜第一版静态内容：

- 可以忽略：单纯追逐“又一个大模型排行榜第一”的新闻。理由：如果没有对应到目标岗位、产品场景或评估方法，短期不需要投入学习。
- 可以忽略：泛泛收集“100 个 AI 工具合集”。理由：工具清单不能直接证明 AI PM 的判断、拆解和评估能力。
- 可以忽略：没有产品落地信息的融资、发布会和概念演示。理由：第一版目标用户更需要训练可交付作品，而不是追全行业动态。
- 30 分钟趋势实验：选一个本周看到的 AI 产品更新，用 5 句话写清楚它解决什么用户问题、AI 介入点是什么、可能失败在哪里、应该看什么指标、能否转成作品片段。

## 3. 能力自测题

第一版使用 5 个能力维度：

- AI 产品判断
- 需求拆解
- 数据与实验意识
- 技术协作理解
- 作品表达能力

每题 4 个选项，分值为 A=0、B=1、C=2、D=3。分值越高，说明该能力越接近可用于面试表达或作品展示。

注意：自测结果只用于筛查当前准备状态，不是能力认证。用户应按自己能在面试中稳定讲清楚的程度选择，而不是选择看起来最完整的答案。

### Q1. 当 JD 写“熟悉 RAG 产品或知识库问答”时，你通常会怎么理解？

| 选项 | 内容 | 分值 |
| --- | --- | --- |
| A | 只知道 RAG 是把资料接给大模型，但说不清细节。 | 0 |
| B | 能解释检索增强生成的大致流程，但不知道怎么评估效果。 | 1 |
| C | 能说出召回、重排、生成、引用来源和常见失败点。 | 2 |
| D | 能把 RAG 拆成用户场景、数据源、检索策略、评估集和反馈闭环。 | 3 |

- 考察维度：AI 产品判断、技术协作理解
- 结果解释：高分用户能把技术名词转成产品方案；低分用户需要先补“RAG 如何影响用户体验和评估指标”。

| 选项 | 内容 | 分值 |

| B | 先找类似产品参考。 | 1 |

| D | 同时判断用户场景、AI 介入必要性、输入输出边界、失败风险和评估方式。 | 3 |

- 考察维度：AI 产品判断、需求拆解
- 结果解释：高分用户会先验证 AI 介入是否必要；低分用户容易把 AI 当成默认解法。

## 4. 计分和分层规则

### 4.1 总分怎么算

- 每题最高 3 分，10 题总分最高 30 分。
- 总分 = 所有题目分值相加。
- 前端静态实现时，只需要为每个选项配置 `score: 0 | 1 | 2 | 3`。

### 4.2 维度分怎么算

每题可以关联 1-2 个能力维度。维度分使用平均分，换算成 100 分制：

```text
维度原始均分 = 该维度关联题目的得分总和 / 该维度关联题目数量
维度分 = round(维度原始均分 / 3 * 100)
```

如果一道题关联两个维度，该题分数同时计入两个维度。第一版不做复杂权重。

### 4.3 等级划分

| 等级 | 总分范围 | 解释 | 建议 |
| --- | --- | --- | --- |
| L1 概念补课期 | 0-10 | 能识别一些 AI 名词，但还不能稳定转成产品判断。 | 先做基础场景拆解和 JD 翻译，不急着写复杂作品。 |
| L2 方案成型期 | 11-18 | 能说出基本方案，但评估指标、失败兜底和技术边界较弱。 | 优先完成一个结构化训练任务，沉淀作品片段。 |
| L3 面试准备期 | 19-25 | 已具备 AI PM 基本判断，作品需要补强证据链和追问准备。 | 做模型评估、Agent 兜底或项目复盘类任务。 |
| L4 作品打磨期 | 26-30 | 能较完整表达场景、技术取舍和评估闭环。 | 重点打磨作品集表达、面试追问和业务价值证明。 |

### 4.4 每个等级对应建议

- L1：推荐 T05、T09、JD 翻译模板；目标是把岗位要求翻译成可理解任务。
- L2：推荐 T01、T02、T06；目标是补齐评估指标和失败兜底。
- L3：推荐 T03、T04、T07、T10；目标是提升技术取舍和面试追问承压能力。
- L4：推荐 T09、T10，并建议提交已有作品进入 Rubric 点评；目标是把作品表达打磨成面试证据。

### 4.5 如何推荐训练任务

第一版使用简单规则：

1. 找出分数最低的能力维度。
2. 如果有多个并列最低，优先级为：数据与实验意识 > 技术协作理解 > AI 产品判断 > 需求拆解 > 作品表达能力。
3. 按下面的静态映射表选择该维度对应任务。
4. 同时结合总等级控制任务难度：
   - L1：推荐基础拆解或表达类任务。
   - L2：推荐评估和兜底类任务。
   - L3：推荐模型选择、Agent 工具调用、失败复盘类任务。
   - L4：推荐作品优化和面试复盘类任务。
5. 结果页展示 1 个主推荐任务 + 2 个备选任务。

| 最低维度 | 主推荐任务 | 备选任务 |
| --- | --- | --- |
| AI 产品判断 | T03 写一份模型选择 memo | T04 为 AI 搜索产品设计真实性与相关性指标、T08 为多模态识别场景设计人工审核机制 |
| 需求拆解 | T05 拆解一个 AI 工作流并判断 AI 介入点 | T02 为客服 Agent 设计失败兜底方案、T06 为 AI 写作产品设计用户反馈闭环 |
| 数据与实验意识 | T01 为企业知识库问答设计 RAG 评估集 | T04 为 AI 搜索产品设计真实性与相关性指标、T10 为一个 AI 产品写失败 case 复盘 |
| 技术协作理解 | T02 为客服 Agent 设计失败兜底方案 | T07 设计一个 Agent 工具调用评估方案、T08 为多模态识别场景设计人工审核机制 |
| 作品表达能力 | T09 优化一段 AI PM 面试项目复盘 | T10 为一个 AI 产品写失败 case 复盘、T05 拆解一个 AI 工作流并判断 AI 介入点 |

## 5. 训练任务库

### T01. 为企业知识库问答设计 RAG 评估集

- 适合等级：L2、L3
- 预计时间：2 小时
- 训练目标：训练用户把“RAG 效果好不好”拆成可验证的评估样本和指标。
- 任务说明：假设一家 B2B SaaS 公司要上线内部知识库问答，用户会询问制度、产品文档、客户案例和操作流程。请设计一套小型 RAG 评估集。
- 用户需要提交什么：10 条测试问题、理想答案要点、引用来源要求、评分维度、失败 case 分类。
- 参考完成标准：覆盖事实型、流程型、模糊型和越权型问题；能区分召回失败和生成失败；包含人工抽检方式。
- 对应能力维度：数据与实验意识、技术协作理解

### T02. 为客服 Agent 设计失败兜底方案

- 适合等级：L2、L3
- 预计时间：2 小时
- 训练目标：训练 Agent 异常路径、人工接管和用户告知设计。
- 任务说明：一个电商客服 Agent 可以查询订单、解释退款规则、发起售后申请。请设计它在工具调用失败、用户意图不清和政策冲突时的兜底方案。
- 用户需要提交什么：失败类型列表、处理流程、用户提示文案、人工接管规则、失败样本回流机制。
- 参考完成标准：至少覆盖工具失败、权限不足、意图不清、政策冲突、用户情绪升级；说明哪些场景必须人工接管。
- 对应能力维度：技术协作理解、需求拆解

### T03. 写一份模型选择 memo

- 适合等级：L3、L4
- 预计时间：2-3 小时
- 训练目标：训练成本、延迟、质量、稳定性之间的产品取舍。
- 任务说明：假设你要为“AI 会议纪要摘要”功能选择模型，请在两个候选模型之间做选择，并说明上线策略。
- 用户需要提交什么：候选模型对比表、评估样本、质量指标、成本估算、延迟要求、降级策略。
- 参考完成标准：不只写“效果更好”，要明确业务场景阈值、评估方法和不可接受风险。
- 对应能力维度：AI 产品判断、数据与实验意识

### T04. 为 AI 搜索产品设计真实性与相关性指标

- 适合等级：L2、L3
- 预计时间：2 小时
- 训练目标：训练搜索类 AI 产品的效果拆解能力。
- 任务说明：假设一个 AI 搜索产品会综合网页和内部文档回答用户问题，请设计真实性、相关性和可用性指标。
- 用户需要提交什么：指标定义、评分样例、测试查询集、人工评审规则、线上反馈入口。
- 参考完成标准：能区分“答案相关但不真实”“答案真实但没解决问题”“引用来源不可信”等情况。
- 对应能力维度：数据与实验意识、AI 产品判断

### T05. 拆解一个 AI 工作流并判断 AI 介入点

- 适合等级：L1、L2
- 预计时间：1.5 小时
- 训练目标：训练用户判断 AI 应该介入流程的哪个环节，而不是把全流程都交给 AI。
- 任务说明：选择一个 AI PM 求职训练场景，例如 AI 作品评分助手、JD 技术要求翻译助手或面试项目复盘助手，拆解用户完成任务的完整流程，并判断哪些环节适合 AI。
- 用户需要提交什么：流程步骤、用户痛点、AI 介入点、非 AI 规则环节、失败风险、最小评估指标。
- 参考完成标准：能说明为什么某些环节不适合 AI 自动化；至少列出 2 个需要人工确认的节点，并说明如何判断 AI 输出是否可用。
- 对应能力维度：需求拆解、AI 产品判断

### T06. 为 AI 写作产品设计用户反馈闭环

- 适合等级：L2、L3
- 预计时间：2 小时
- 训练目标：训练从用户反馈到评估集更新的闭环意识。
- 任务说明：一个 AI 写作产品支持生成小红书标题、公众号开头和邮件草稿。请设计用户反馈如何进入产品迭代。
- 用户需要提交什么：反馈入口、反馈标签、日志字段、人工审核规则、评估集更新流程。
- 参考完成标准：反馈不只停留在点赞/点踩；能说明如何把失败样本变成下一轮评估材料。
- 对应能力维度：数据与实验意识、需求拆解

### T07. 设计一个 Agent 工具调用评估方案

- 适合等级：L3、L4
- 预计时间：2-3 小时
- 训练目标：训练用户评估 Agent 是否正确选择工具、正确传参和正确处理结果。
- 任务说明：假设一个旅行 Agent 可以查航班、查酒店、生成行程。请设计工具调用评估方案。
- 用户需要提交什么：任务样本、工具选择标准、参数正确性检查、成功率指标、失败类型。
- 参考完成标准：能区分“工具选错”“参数错”“工具结果正确但总结错误”“任务本身不可完成”。
- 对应能力维度：技术协作理解、数据与实验意识

### T08. 为多模态识别场景设计人工审核机制

- 适合等级：L2、L3
- 预计时间：2 小时
- 训练目标：训练多模态场景中的风险控制和人工审核设计。
- 任务说明：假设一个平台要用图片识别辅助审核商品违规内容。请设计模型判断不确定时的人审机制。
- 用户需要提交什么：风险分级、置信度处理、人工审核队列、误判反馈、用户申诉路径。
- 参考完成标准：能说明哪些结果可自动通过、哪些必须人审、哪些需要二次复核。
- 对应能力维度：AI 产品判断、技术协作理解

### T09. 优化一段 AI PM 面试项目复盘

- 适合等级：L1、L4
- 预计时间：1.5 小时
- 训练目标：训练作品表达，把经历改写成能被面试官追问和验证的作品片段。
- 任务说明：选择一个你做过或模拟的 AI 产品项目，把“我做了什么”改写成“我如何判断、取舍、验证和复盘”。
- 用户需要提交什么：项目背景、用户问题、方案取舍、指标设计、失败复盘、面试表达版本。
- 参考完成标准：少写功能堆叠，多写关键判断；至少准备 5 个面试追问回答。
- 对应能力维度：作品表达能力、AI 产品判断

### T10. 为一个 AI 产品写失败 case 复盘

- 适合等级：L3、L4
- 预计时间：2 小时
- 训练目标：训练用户从失败样本中提取产品改进和评估规则。
- 任务说明：选一个 AI 产品失败场景，例如答非所问、幻觉、工具调用失败或用户不采纳，写一份失败 case 复盘。
- 用户需要提交什么：失败现象、影响用户、原因假设、验证方式、修复方案、后续指标。
- 参考完成标准：不能只写“模型不好”；要说明问题定位路径、短期兜底和长期迭代机制。
- 对应能力维度：数据与实验意识、作品表达能力

## 6. JD 技术要求翻译模板

### 6.1 输入格式

第一版只保留必要输入：

- JD 原文：用户粘贴 1-3 段岗位描述或技术要求。
- 目标身份：校招生 / 实习生 / 0-3 年 PM / 转岗者。
- 目标方向：AI 应用 / RAG / Agent / 策略产品 / 增长产品 / 暂不确定。

### 6.2 输出结构

JD 翻译结果按 5 段输出：

1. 岗位能力画像：这段 JD 通常对应哪些能力要求。
2. 技术关键词解释：把 RAG、Agent、模型评估、数据闭环等词翻译成人话。
3. 可能追问方向：面试官可能如何追问。
4. 作品集缺口：用户需要补哪类作品证据。
5. 7 天练习建议：推荐 1 个主任务和 1 个轻量补充动作。

### 6.3 示例 JD 片段

```text
负责 AI 知识库问答产品的需求分析和效果优化，熟悉 RAG、向量检索、模型评估方法，能够与算法和工程团队协作，推动问答准确率、召回率和用户满意度提升。
```

### 6.4 示例翻译结果

**岗位能力画像**

这段 JD 通常不只是要求你知道 RAG 名词，也可能要求你能把知识库问答拆成数据源、检索、生成、引用、评估和反馈闭环，并和研发一起定位效果问题。
这只是基于 JD 文本的准备方向，不代表该公司的内部面试标准。

**技术关键词解释**

| JD 关键词 | 用户需要理解成什么 |
| --- | --- |
| RAG | 大模型回答前先从资料库找证据，再基于证据生成答案。产品侧要关注资料质量、召回结果、答案可信度和引用来源。 |
| 向量检索 | 把文本转成向量后做相似度匹配。产品侧不需要推公式，但要知道它会影响“能不能找对资料”。 |
| 模型评估方法 | 不是只看用户喜不喜欢，而是要设计测试问题、理想答案、评分规则和失败样本分析。 |
| 准确率 / 召回率 | 准确率关注找到的内容对不对，召回率关注该找到的内容有没有漏。实际产品里还要看用户是否解决问题。 |

**可能追问方向**

- 你如何判断一次错误回答是检索问题还是生成问题？
- 你会如何构造一个小型 RAG 评估集？
- 如果准确率提高但回答速度变慢，你会怎么取舍？
- 用户反馈“答案不可信”时，你会改产品还是改模型？

**作品集缺口**

你需要补一份能证明“评估方法”的作品，而不是只写一个知识库问答 PRD。最小作品可以是：10 条测试问题 + 理想答案 + 评分规则 + 失败 case 分类 + 迭代建议。

**7 天练习建议**

- 主任务：T01 为企业知识库问答设计 RAG 评估集。
- 轻量动作：找一个公开帮助中心或产品文档，手工写 10 个用户问题，并标注每个问题需要引用的资料来源。

### 6.5 风险提示

- JD 翻译结果只基于用户粘贴内容和公开通用知识，不代表公司内部招聘标准。
- 第一版不自动抓取招聘网站，不保存 JD，不采集隐私信息。
- 如果 JD 涉及公司未公开项目、面试题或内部资料，用户不应粘贴。
- 翻译结果用于训练和准备面试，不等同于录用建议。

## 8. 作品评分 Rubric

第一版使用 5 个评分维度，每个维度 1-5 分。该 Rubric 适合人工或半人工点评；第一版不假装能全自动精准评分。

### 8.1 问题定义与用户场景

| 分数 | 描述 |
| --- | --- |
| 1 | 只描述功能，没有明确用户、场景和问题。 |
| 2 | 有背景，但用户痛点模糊，问题重要性不足。 |
| 3 | 能说明目标用户、具体场景和主要问题。 |
| 4 | 能说明问题频率、影响范围和现有方案缺口。 |
| 5 | 问题定义清晰，有真实约束、优先级判断和可验证目标。 |

### 8.2 AI 介入必要性与技术边界

| 分数 | 描述 |
| --- | --- |
| 1 | 默认用 AI，没有解释为什么需要 AI。 |
| 2 | 能说明 AI 可能有用，但边界和失败风险不清楚。 |
| 3 | 能说明 AI 适合处理的环节和不适合的环节。 |
| 4 | 能识别模型能力、数据质量、成本、延迟和可靠性约束。 |
| 5 | 能给出 AI / 规则 / 人工协作边界，并说明取舍理由。 |

### 8.3 方案拆解与协作可落地性

| 分数 | 描述 |
| --- | --- |
| 1 | 方案停留在想法层，没有流程和角色分工。 |
| 2 | 有基础流程，但输入、输出和异常路径不完整。 |
| 3 | 能拆出核心流程、关键页面或关键交互。 |
| 4 | 能说明数据、模型、工程、运营或审核如何协作。 |
| 5 | 方案可以指导研发评估工作量，并包含上线、灰度或运营机制。 |

### 8.4 评估指标与反馈闭环

| 分数 | 描述 |
| --- | --- |
| 1 | 没有指标，或只有“用户满意度”这类泛指标。 |
| 2 | 有指标，但不可验证或和问题关系弱。 |
| 3 | 有 2-3 个关键指标，并能说明如何收集。 |
| 4 | 包含离线评估、线上反馈、失败 case 和版本对比。 |
| 5 | 能把用户反馈、人工标注、评估集更新和产品迭代连成闭环。 |

### 8.5 作品表达与面试说服力

| 分数 | 描述 |
| --- | --- |
| 1 | 像功能说明，不像作品集案例。 |
| 2 | 能说明自己做了什么，但缺少判断和取舍。 |
| 3 | 能表达背景、方案、指标和个人贡献。 |
| 4 | 能突出关键决策、失败复盘和面试追问准备。 |
| 5 | 能把作品讲成“我如何判断、验证、协作和迭代”的能力证据。 |

### 8.6 评分输出规则

- 总分 = 5 个维度得分相加，最高 25 分。
- 20-25 分：可进入作品集打磨。
- 15-19 分：有作品雏形，需要补评估、边界或表达。
- 10-14 分：需要重写结构，先补问题定义和方案拆解。
- 5-9 分：暂不建议作为面试作品，先完成一个训练任务。

第一版反馈页按点评状态输出：

- 待人工点评：只展示提交成功状态、Rubric 自查结构、通用修改建议和通用面试追问，不展示总分和分项分。
- 已人工点评：展示总分、分项分、最大短板、三条优先修改建议、五个面试追问和下一步推荐训练任务。

## 10. 第一版不做的内容

第一版明确不做：

- 大规模题库。
- 个性化长期学习路径。
- 自动抓取 JD。
- 自动生成完整作品。
- 用户账号体系。
- 登录、注册和历史记录。
- 数据库和后台管理系统。
- 支付课程、付费内容或训练营交付。
- 自动化 AI 精准评分。
- 公开作品案例库。
- 自动抓取 AI 新闻或热点。
- 企业培训后台。
- 微信小程序或原生 App。

第一版允许轻量保留：

- 静态自测题。
- 静态训练任务。
- 静态 JD 翻译模板。
- 静态忽略榜说明。
- 半人工 Rubric 点评流程。
