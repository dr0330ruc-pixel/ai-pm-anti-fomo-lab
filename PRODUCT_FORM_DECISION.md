# AI PM 技术评估训练工作台：产品形态决策

> 版本：v0.1  
> 日期：2026-06-07  
> 当前结论：第一版做网页 / H5，不做微信小程序，不做原生 App，不自建云服务器。

## 1. 决策结论

第一版产品形态确定为：

> 网页 / H5 优先，桌面端体验优先，同时兼容移动端浏览。

第一版按当前 `PRD.md`、`MVP_SCOPE.md`、`USER_FLOW.md` 推进，不再继续砍 MVP 范围。

需要注意的是，当前版本不是极限压缩的最小 MVP，而是：

> 偏完整的 v0.1 网页原型。

它的目标是让目标用户完整感受到“自测 -> 训练任务 -> 作品评分 -> 面试表达建议”的产品闭环，同时展示 JD 翻译和本周忽略榜这两个差异化入口。

## 2. 为什么选择网页 / H5

### 2.1 更适合核心使用场景

本产品的核心动作包括：

- 粘贴 JD。
- 填写能力自测。
- 阅读较长的能力结果和训练任务。
- 提交作品片段。
- 查看 Rubric 点评和面试追问。

这些动作更适合网页，尤其是桌面端。

### 2.2 更适合快速迭代

网页 / H5 可以更快完成：

- 页面调整。
- 内容更新。
- 静态数据替换。
- 部署上线。
- 分享给测试用户。

第一版需要快速验证用户是否愿意自测、训练和提交作品，不适合被小程序审核、平台限制和登录体系拖慢。

### 2.3 更适合内容传播

本产品未来的传播渠道包括：

- 公众号。
- 即刻。
- 小红书。
- AI PM 社群。
- 求职社群。
- 产品经理社群。

网页链接更适合跨平台传播和快速打开。

### 2.4 更适合后续扩展

网页应用后续可以平滑扩展：

- AI API 分析。
- 用户账号。
- 数据库存储。
- 作品案例库。
- 支付。
- 管理后台。
- SEO 内容页。

## 3. 为什么第一版不做微信小程序

微信小程序暂不作为第一版首发形态。

原因：

- 粘贴长 JD 和长作品体验较弱。
- 查看长文本反馈不如网页自然。
- 小程序审核和发布流程会降低迭代速度。
- 第一版还没有验证用户是否愿意提交作品，不适合先投入小程序工程。
- 小程序更适合作为后续分发渠道，而不是 MVP 主产品形态。

后续可以在以下条件满足后再考虑小程序：

- 已经有稳定用户来源。
- 用户需要移动端持续访问。
- 已经验证作品点评或训练任务有付费价值。
- 产品核心路径稳定。

## 4. 为什么第一版不做原生 App

原生 App 暂不考虑。

原因：

- 开发成本高。
- 分发成本高。
- 用户安装门槛高。
- 当前需求不依赖系统级能力。
- MVP 阶段不需要原生性能或推送能力。

## 5. 为什么第一版不自建云服务器

第一版暂不自建云服务器。

原因：

- 当前核心目标是验证用户路径，不是验证后端架构。
- 自测题、训练任务、Rubric、忽略榜内容都可以先用静态数据。
- 作品点评早期可以半人工处理。
- 前端托管即可满足第一版展示和试用。

优先选择：

- Vercel。
- Cloudflare Pages。
- Netlify。

后续只有出现以下需求时，再评估后端和云服务器：

- 需要保存用户自测历史。
- 需要管理作品提交记录。
- 需要用户登录。
- 需要 AI API 调用额度控制。
- 需要付费。
- 需要公开作品点评案例库。
- 需要后台审核和运营工具。

## 6. 第一版保留的产品范围

第一版继续保留当前 MVP 文档中的主要范围。

### 6.1 P0 功能

必须完成：

- 工作台首页。
- 能力自测页。
- 自测结果页。
- 训练任务列表页。
- 训练任务详情页。
- 作品评分页。

### 6.2 P1 功能

轻量完成：

- JD 技术要求翻译页。
- 本周可以忽略榜页面。

P1 功能不追求完整自动化，优先用静态内容、规则模板或示例结果跑通体验。

## 7. 第一版实现策略

### 7.1 静态数据优先

第一版可以使用静态数据完成：

- 6 个能力维度。
- 自测题库。
- 自测评分规则。
- 10 个训练任务。
- 作品评分 Rubric。
- 示例 JD。
- JD 翻译规则或模板。
- 本周可以忽略榜内容。
- 30 分钟趋势实验。

### 7.2 规则和模板优先

第一版不强求 AI 自动分析。

可以先用规则和模板完成：

- 自测评分。
- 推荐训练任务。
- JD 技术关键词识别。
- 作品评分结构展示。
- 面试追问示例。

### 7.3 作品点评半人工优先

作品评分器第一版不假装完全自动化。

推荐实现方式：

1. 页面展示 Rubric。
2. 用户提交作品片段。
3. 系统给出基础提示和提交成功反馈。
4. 主理人根据 Rubric 半人工点评。
5. 后续把真实点评沉淀为案例和评分样本。

这样可以优先保证反馈质量，并用真实用户作品校准评分标准。

## 8. 第一版不是极限 MVP

当前版本保留了较完整的产品形态，因此需要接受以下取舍：

- 开发周期可能超过最小 MVP。
- 内容准备压力更大。
- 10 个训练任务需要时间打磨。
- JD 翻译和忽略榜第一版只能轻量实现。
- 作品评分的真实价值依赖半人工点评质量。

对应策略：

- 先保证主路径能跑通。
- P1 页面允许内容静态化。
- 不为 P1 功能引入复杂后端。
- 不因为当前范围较大而提前做登录、支付或数据库。

## 9. 第一版完成标准

第一版网页原型完成时，应满足：

- 用户能从首页进入能力自测。
- 用户能完成自测并看到六维结果。
- 用户能看到最大短板和推荐任务。
- 用户能浏览 10 个训练任务。
- 用户能打开任务详情并知道要交付什么。
- 用户能进入作品评分页。
- 用户能提交或模拟提交作品。
- 用户能看到 Rubric、修改建议结构和面试追问。
- 用户能进入 JD 翻译页并看到示例分析结果。
- 用户能进入本周可以忽略榜并理解反FOMO定位。
- 页面可以部署成一个公网可访问链接。
- 移动端可阅读，桌面端可高效填写。

## 10. 当前决策状态

本决策确认：

- 第一版做网页 / H5。
- 不做微信小程序首发。
- 不做原生 App。
- 不自建云服务器。
- 不继续砍当前 MVP 范围。
- P0 功能完整做。
- P1 功能轻量做。
- 静态数据、规则模板、半人工点评优先。
